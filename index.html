DeadRinger

Windows console program that wraps legacy scripts and other console programs.
Code on GitHub ZIP TAR

Why should you care?

Many (especially 3rd party) Windows scripts and console programs do not write to the Windows event log or even to a log file. This of course makes debugging difficult if, for example, you are running a script on thousands of endpoints via Group Policy. You have to hope that it completes successfully.

DeadRinger is here to help!

DeadRinger writes the stdout and stderr streams of your legacy program or script to the Windows event log. You can also configure DeadRinger to write the result of the streams to many other targets (e.g. email, remote server, file). Thanks Apache log4net!

But wait, there is more. DeadRinger can also delay execution of the app it wraps by some amount of time you specify. This makes sense to do if you desire to stagger the execution of your script on multiple endpoints, but don't really care exactly when it runs.

How does it work?

DeadRinger log messages will appear in the Windows event log as "DeadRinger" events. If you are running DeadRinger for the first time on an endpoint, you must run it with admin privileges. This causes the "DeadRinger" source to be created in the event log. DeadRinger events will not appear in the event log if you do not do this.

Basic usage looks like this:

C:\>DeadRinger.exe -pC:\foo\bar.exe

With a random execution delay of up to 15 minutes:

C:\>DeadRinger.exe -pC:\foo\bar.exe -d900000

With optional arguments for the wrapped app (bar.exe in this case):

C:\>DeadRinger.exe -pC:\foo\bar.exe -a"--baz=value1 --qux=value2"

How do I log to other targets?

This is where Apache log4net comes into play. You will need to add the appropriate appender to DeadRinger's App.config file. The log4net documentation has many config examples for a variety of appenders.
